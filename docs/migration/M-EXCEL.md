# M-EXCEL — Migration runbook: generic Excel / CSV / paper → SmartERP

- **Runbook id**: M-EXCEL
- **Source vendor**: n/a (Microsoft Excel + Microsoft 365 + paper)
- **Source product**: Excel workbook(s) per company; sometimes paper-only ledgers transcribed during the migration.
- **Source-vendor version coverage**: Excel 2016+ (.xlsx), Microsoft 365, Google Sheets exports (.xlsx + .csv).
- **Status**: First-production-cutover-eligible 2026-Q3.
- **Engineer owner**: Migration team lead.
- **Customer-success owner**: CS lead Verona.
- **Last reviewed**: 2026-04-28.
- **Plan reference**: §21.1 + ADR-040.

## 1. Source system overview

### 1.1 Why Excel matters

Excel is the most common "source system" in the smallest tier of the SmartERP target market — single-handler artisan firms, very-small commercio, sub-€500k-revenue manifatturiere. The data quality is variable; the migration runbook standardises a workbook structure the customer fills in (or the migration engineer fills in during a paper-to-Excel transcription).

### 1.2 Supported workbook structure

The runbook ships a **template workbook** at `docs/migration/M-EXCEL/template.xlsx` (referenced; not embedded in this markdown). The template carries 8 worksheets, each with a fixed column header row at row 1 and data starting at row 2.

| Sheet | Domain | Required for cutover |
|---|---|---|
| `Anagrafica_Clienti` | Customers | yes |
| `Anagrafica_Fornitori` | Suppliers | only if procurement module in scope |
| `Articoli` | Products | yes |
| `Documenti_Vendita` | Sales orders + DDTs + invoices | yes |
| `Righe_Documento` | Document lines | yes |
| `Documenti_Acquisto` | Supplier invoices | only if AP in scope |
| `Magazzino_Apertura` | Stock opening | only if inventory in scope |
| `Apertura_Saldi` | AR / AP opening balances | yes |

Each sheet has a sample-row (hidden in the template) showing the expected format.

### 1.3 Known quirks

- Date formats: Excel auto-formats `01/02/2026` ambiguously (US: Jan 2 vs IT: 1 Feb). The template forces ISO `YYYY-MM-DD` text in date cells; the validator rejects ambiguous formats.
- Merged cells: refused. The template documents the constraint and the validator flags merged-cell rows.
- Currency formatting: amounts are stored as numbers in EUR; the validator rejects strings.
- Customer codes: when blank, the importer auto-generates `C-0001`, `C-0002`, … per tenant.

## 3. Data export procedure

### 3.1 Customer-side fill

The customer (or the migration engineer during a transcription session) populates the template workbook. The engineer holds a 30-min Zoom call with the customer's data steward to demonstrate each sheet. A 1-hour follow-up Zoom checks the workbook before upload.

### 3.2 Validation of the workbook

```bash
node dist/scripts/migration/M-EXCEL/validate-workbook.js \
  --input ./uploads/<ticket>/template.xlsx \
  --report ./uploads/<ticket>/validation-report.json
```

The validator checks:

- header row matches the template exactly,
- no merged cells,
- all date cells parse as ISO `YYYY-MM-DD`,
- partita IVA + codice fiscale checksum,
- per-sheet referential integrity (every `Righe_Documento.documentId` references a `Documenti_Vendita.id` row),
- IVA rates ∈ {0, 4, 5, 10, 22} (warning, not error, on other values),
- duplicate customer codes within `Anagrafica_Clienti`.

### 3.3 Staging tenant load

```bash
node dist/scripts/migration/M-EXCEL/import-workbook.js \
  --tenant <stagingTenantId> \
  --input ./uploads/<ticket>/template.xlsx \
  --dry-run
```

## 4. Mapping

The template's column headers are the canonical mapping; this section makes it explicit.

### 4.1 `Anagrafica_Clienti`

| Column | SmartERP `customers` field | Notes |
|---|---|---|
| Codice | `code` | autogenerated when blank. |
| Ragione_Sociale | `name` | trimmed. |
| Partita_IVA | `vatNumber` | as-is. |
| Codice_Fiscale | `fiscalCode` | as-is. |
| Indirizzo | `address` | as-is. |
| CAP | `postalCode` | as-is. |
| Citta | `city` | as-is. |
| Provincia | `province` | as-is. |
| Nazione_ISO2 | `country` | default `IT`. |
| PEC | `pecEmail` | as-is. |
| Email | `email` | as-is. |
| Telefono | `phone` | as-is. |
| Codice_Destinatario_SDI | `sdiDestinationCode` | default `0000000`. |
| IVA_Default_PCT | `defaultIvaRate` | default 22. |
| Termini_Pagamento_GG | `paymentTermsDays` | default 30. |
| Split_Payment | `splitPayment` | boolean; default false. |

### 4.2 `Articoli`

| Column | SmartERP `products` field | Notes |
|---|---|---|
| Codice_Articolo | `sku` | required. |
| Descrizione | `description` | trimmed. |
| Unita_Misura | `unitOfMeasure` | enum-mapped. |
| Categoria | `categoryCode` | as-is. |
| EAN | `barcode` | as-is. |
| Prezzo_Vendita_Default | `defaultUnitPriceCents` | euros → cents (R-D04). |
| IVA_PCT | `defaultIvaRate` | as-is. |
| NC8_Intrastat | `intrastatNc8` | optional. |
| Massa_Netta_Kg | `netMassKg` | optional. |
| Paese_Origine | `countryOfOrigin` | optional. |

### 4.3 `Documenti_Vendita`

| Column | SmartERP target | Notes |
|---|---|---|
| Tipo_Documento | `documentType` (TD01/…) for invoices; mapped to `sales_orders` / `ddts` for non-invoice types. |
| Numero | `number` | as-is. |
| Anno | `fiscalYear` | as-is. |
| Data | `invoiceDate` / `orderDate` / `issueDate` | ISO date. |
| Codice_Cliente | `customerId` | resolved via `customers.code`. |
| Subtotale | `subtotalAmount` | EUR. |
| IVA | `taxAmount` | EUR. |
| Totale | `totalAmount` | EUR. |
| Note | `notes` | as-is. |

### 4.4 `Righe_Documento`

Per-line decomposition referenced from `Documenti_Vendita.id`. Columns: `Documento_Id`, `Codice_Articolo`, `Descrizione`, `Quantita`, `Prezzo_Unitario`, `IVA_PCT`, `Sconto_PCT`, `Natura_IVA` (when N1..N7 applies), `Totale_Riga`.

### 4.5 `Documenti_Acquisto`

Same shape as §4.3 but routed into `supplier_invoices`. The `partnerCountry` + `partnerVatNumber` columns enable INTRA-2bis aggregation.

### 4.6 `Magazzino_Apertura`

| Column | SmartERP target |
|---|---|
| Codice_Articolo | `stock_movements.productId` (resolved via `products.sku`) |
| Magazzino_Codice | `stock_movements.warehouseId` (resolved via `warehouses.code`) |
| Quantita_Apertura | `stock_movements.quantity` (as `opening_balance` movement type) |
| Costo_Medio_Cents | unit cost cents — populates the warehouse's last-cost cache |

### 4.7 `Apertura_Saldi`

Per-partner residual:

| Column | SmartERP target |
|---|---|
| Tipo (`AR`/`AP`) | discriminates customer vs supplier |
| Codice_Partner | `customers.code` or supplier UUID |
| Importo_Residuo | residual cents |
| Data_Origine | original document date |
| Riferimento_Documento | source-document reference (preserved in `notes`) |

## 5. Validation

### 5.1 Dry-run report contract

Same shape as M-TS-LYNFA §5.1 with `runbookId = "M-EXCEL"`.

Acceptance: zero `blockers`; anomalies under `2%` of the parent table row count (Excel data quality is acceptable when the template is followed; higher anomaly rate triggers a "go back to the customer for fixes" loop before cutover).

### 5.2 Sample-customer sign-off

Same as M-TS-LYNFA §5.2 + a manual review of the chart of accounts (the customer's commercialista almost always wants to refine the Piano dei Conti seed).

## 6. Cutover-day checklist

```
T-30d: scope + template delivered; customer fills it
T-21d: customer returns the filled workbook; engineer validates
T-14d: trial migration into staging tenant; commercialista review
T-7d : cutover-day rehearsal in staging
T-1d : final workbook delivered with last-week deltas

T-day Europe/Rome:
  06:00 — Customer sends final workbook (incremental delta from T-7d)
  07:00 — Engineer runs validate-workbook.js
  08:00 — Engineer runs import-workbook.js dry-run; reviews report
  09:00 — Engineer runs import-workbook.js against production tenant
  10:00 — Customer-side spot check on 10 sampled documents
  11:00 — Commercialista spot check on IVA + Piano dei Conti
  12:00 — Lunch (no migrations)
  13:00 — Magazzino opening + Apertura saldi loaded
  14:00 — Engineer runs post-migration validation report
  15:00 — Customer-side go / no-go
  16:00 — If GO: SmartERP turns on; the Excel workbook becomes a read-only archive on the customer's filer
  17:00 — Customer-success closes the cutover ticket; sign-off PDFs collected
```

A no-go at 15:00 triggers the rollback procedure.

## 7. Rollback procedure

The Excel workbook source is by definition still available (the customer never lost it). Rollback within 24h:

1. SmartERP team de-activates the tenant.
2. The customer reverts to using their Excel workbook for new data.
3. Any data entered in SmartERP since cutover is exported as CSV and the customer manually copies it back into the workbook.
4. Post-mortem written.

## 8. Post-migration validation

Same as M-TS-LYNFA §8 with one extra check at T+30: the commercialista signs the first IVA liquidation, *and* the customer's data steward confirms the per-customer count parity (Excel doesn't auto-count; the parity check is manual).
