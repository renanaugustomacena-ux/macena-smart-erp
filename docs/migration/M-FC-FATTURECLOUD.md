## 1. Source system overview

### 1.1 What Fatture in Cloud is

Fatture in Cloud (FiC) is a SaaS gestionale focused on small businesses, professionals, and the sub-10-employee tier. TeamSystem acquired it in 2018; it remains the single highest-volume source system in the Italian micro-SME segment. The FiC API v2 is well-documented (`developers.fattureincloud.it`) and supports OAuth 2.0 with a per-customer token.

### 1.2 Deployment model

Pure SaaS — no on-premise component. Customer logs in at `secure.fattureincloud.it`. Data lives in TeamSystem's cloud; export is via the API or via the in-app CSV export.

### 1.3 Key API resources

| FiC resource | Domain | SmartERP target |
|---|---|---|
| `GET /c/{company_id}/clients` | Customers | `customers` |
| `GET /c/{company_id}/suppliers` | Suppliers | `suppliers` UUID |
| `GET /c/{company_id}/products` | Products | `products` |
| `GET /c/{company_id}/issued_documents?type=invoice` | Active invoices | `invoices` (TD01) |
| `GET /c/{company_id}/issued_documents?type=delivery_note` | DDT | `ddts` |
| `GET /c/{company_id}/received_documents?type=invoice` | Supplier invoices | `supplier_invoices` |
| `GET /c/{company_id}/cashbook` | Cashbook entries | opening AR/AP balances |

### 1.4 Known quirks

- FiC's "regime forfettario" customers carry zero IVA on every line; the importer maps these to Natura `N2.2`.
- FiC's `numeration` field is per-numbering-series. Customers commonly use multiple series (e.g., `FT/2026/`, `NC/2026/`); the importer preserves all series and distinguishes them via the `notes.numerationSeries` field.
- FiC's customer-side "codice destinatario" is sometimes blank for customers who only use PEC. The importer keeps `0000000` as the default.
- FiC's API rate limit is 30 requests/minute/customer-token. The importer paces accordingly; full export of an active company takes 20-90 minutes depending on data volume.

## 3. Data export procedure

### 3.1 OAuth handshake

The customer logs in to FiC, opens "Strumenti → API & Webhooks → Sviluppatori", and authorises the SmartERP migration app. The OAuth scope requested:

- `entity.clients:r`
- `entity.suppliers:r`
- `entity.products:r`
- `issued_documents.invoices:r`
- `issued_documents.delivery_notes:r`
- `received_documents.invoices:r`
- `cashbook:r`

Tokens last 60 minutes; the importer refreshes via the OAuth `refresh_token` grant.

### 3.2 Pull-into-staging

The migration engineer runs:

```bash
node dist/scripts/migration/M-FC-FATTURECLOUD/pull-fic.js \
  --client-id <oauth_client_id> \
  --client-secret <env:FIC_CLIENT_SECRET> \
  --refresh-token <env:FIC_CUSTOMER_REFRESH_TOKEN> \
  --company-id <fic_company_id> \
  --output ./uploads/<ticket>/
```

The script writes one JSON file per FiC resource (`fic_clients.json`, `fic_products.json`, etc.) into the upload directory. Rate-limit-respecting paging is built in.

### 3.3 Validation of the export

`node dist/scripts/migration/M-FC-FATTURECLOUD/validate-export.js --input ./uploads/<ticket>/` reports:

- partita IVA + codice fiscale checksum failures,
- invoice rows with missing `documentType` (FiC sometimes omits TD06 / TD24 codes for niche cases),
- regime forfettario flags with non-zero IVA (data inconsistency in the source).

### 3.4 Staging tenant load

```bash
node dist/scripts/migration/M-FC-FATTURECLOUD/import-fic.js \
  --tenant <stagingTenantId> \
  --input ./uploads/<ticket>/ \
  --dry-run
```

Produces the dry-run report (§5).

## 4. Mapping

### 4.1 Customer (`fic_clients.json`)

| FiC field | SmartERP `customers` field | Notes |
|---|---|---|
| `id` | — (not stored) | preserved in `notes.legacyFicId` for traceability. |
| `code` | `code` | when missing, autogenerated as `C-<sequence>`. |
| `name` | `name` | trimmed. |
| `vat_number` | `vatNumber` | as-is. |
| `tax_code` | `fiscalCode` | as-is. |
| `address_street` + `address_postal_code` + `address_city` + `address_province` + `country_iso` | `address`, `postalCode`, `city`, `province`, `country` | as-is. |
| `pec_email` | `pecEmail` | as-is. |
| `email` | `email` | as-is. |
| `phone` | `phone` | as-is. |
| `e_invoice` (boolean) + `ei_code` | `sdiDestinationCode` | `0000000` when both blank. |
| `default_payment_terms` | `paymentTermsDays` | as-is. |
| `default_vat_rate` | `defaultIvaRate` | as-is. |

### 4.2 Product (`fic_products.json`)

| FiC field | SmartERP `products` field | Notes |
|---|---|---|
| `code` | `sku` | as-is. |
| `name` + `description` | `description` | concatenated when both present. |
| `measure` | `unitOfMeasure` | enum-mapped. |
| `barcode` | `barcode` | as-is. |
| `default_vat.value` | `defaultIvaRate` | as-is. |
| `category` | `categoryCode` | as-is. |

### 4.3 Active invoices (`fic_issued_documents.json` where `type=invoice`)

| FiC field | SmartERP `invoices` field | Notes |
|---|---|---|
| `type_label` (TD01 / TD06 / TD24 / …) | `documentType` | preserved 1:1; LOSSY when FiC reports a non-AdE code. |
| `numeration` + `number` + `year` | `(notes.numerationSeries, number, fiscalYear)` | series preserved in `notes`; M-FC supports per-series numbering. |
| `entity` ↔ `customers.code` | `customerId` | join by FiC client `id` in `notes.legacyFicId`. |
| `date` | `invoiceDate` | as-is. |
| `amount_net` | `subtotalAmount` | as-is. |
| `amount_vat` | `taxAmount` | as-is. |
| `amount_gross` | `totalAmount` | as-is. |
| `items_list[]` | `lines[]` jsonb | flattened — description, qty, unitPrice, ivaRate, naturaCode (TD06/TD24 path). |

### 4.4 DDTs (`fic_issued_documents.json` where `type=delivery_note`)

Maps to `ddts` + `ddt_lines`. **LOSSY**: FiC's "Causale di Trasporto" is free-text; the importer maps the most common values (`vendita`, `reso`, `conto lavorazione`) and flags the rest for manual review.

### 4.5 Supplier invoices (`fic_received_documents.json`)

Maps to `supplier_invoices`. The `partnerCountry` + `partnerVatNumber` (S16.2 schema) are populated from FiC's `entity.country_iso` + `entity.vat_number` when the supplier is foreign — enables INTRA-2bis aggregation immediately.

### 4.6 Cashbook → opening balances

`fic_cashbook.json` rows with `kind = 'in'` (incoming) feed open AR; `kind = 'out'` (outgoing) feed open AP. The migration engineer reviews the balance per partner with the customer's data steward.

## 6. Cutover-day checklist

```
T-30d: scope + OAuth handshake + sample uploaded + mapping v1 signed
T-14d: trial migration into staging tenant; commercialista review
T-7d : cutover-day rehearsal; rollback rehearsed
T-1d : pre-cutover pull (the FiC API is incrementally pulled the morning of T-day)

T-day Europe/Rome:
  06:00 — Source freeze (customer stops issuing in FiC)
  06:30 — Final pull-fic.js run (all resources)
  07:30 — Engineer runs validate-export.js + import-fic.js (dry-run)
  09:00 — Engineer runs import-fic.js against the production tenant
  10:00 — Customer-side spot check on 10 sampled invoices
  11:00 — Commercialista spot check on IVA + prima-nota
  12:00 — Lunch (no migrations)
  13:00 — Cashbook-derived opening balances loaded
  14:00 — Engineer runs post-migration validation report
  15:00 — Customer-side go / no-go
  16:00 — If GO: SmartERP turns on; FiC remains read-only access for the customer (FiC retains the data; the customer's plan downgrades to a read-only tier)
  17:00 — Customer-success closes the cutover ticket; sign-off PDFs collected
```

A no-go at 15:00 triggers the rollback procedure.

## 7. Rollback procedure

Within 24 hours:

1. SmartERP team de-activates the tenant.
2. The customer keeps using FiC (the source remains live throughout the migration window).
3. Any new data entered in SmartERP between cutover and rollback is exported as CSV and the customer manually re-enters it in FiC.
4. Post-mortem written.

After the 24-hour window the customer is fully on SmartERP and the FiC plan downgrades to read-only or cancellation per their FiC contract.
